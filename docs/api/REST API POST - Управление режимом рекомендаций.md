## Описание

| Параметр | Значение |
|----------|----------|
| **URL\*** | `/internal/api/v1/recommendations/mode` |
| **Метод HTTP\*** | `POST` |
| **Наименование метода\*** | Ручное управление режимом сервиса рекомендаций: принудительное включение резервного режима (kill switch) и возврат к автоматическому управлению circuit breaker |
| **Способ авторизации\*** | JWT (token) сотрудника, внутренний контур |
| **Роли\*** | `sre`, `on-call-engineer` (внутренняя ролевая модель эксплуатации) |
| **Ссылка на swagger** | – |
| **http-статус коды в ответе** | 200 – ок; Ошибки: 400, 401, 403, 500 |

## Входные параметры\*

| Параметр | Тип | Описание | Расположение | Обязательный | Значение по умолчанию | Пример |
|----------|-----|----------|--------------|--------------|----------------------|--------|
| authorization | string | JWT (token) сотрудника | Header | Да | | |
| mode | string | Целевой режим управления: `degraded` – принудительно включить резервный режим (kill switch); `auto` – вернуть автоматическое управление circuit breaker | Body | Да | | `degraded` |
| reason | string | Причина переключения; попадает в журнал переключений и уведомление дежурной смене | Body | Да | | `Плановые работы на ML-контуре` |

## Пример запроса

*В разделе приводится пример выполняемого запроса.*

```json
{
  "mode": "degraded",
  "reason": "Плановые работы на ML-контуре до 18:00"
}
```

## Выходные параметры\*

| Параметр | Тип | Описание | Обязательный | Значение по умолчанию | Пример |
|----------|-----|----------|--------------|----------------------|--------|
| mode | string | Текущий действующий режим системы после выполнения запроса: `NORMAL` / `DEGRADED` / `RECOVERY` | Да | | `DEGRADED` |
| control | string | Способ управления режимом: `manual` (kill switch активен) / `auto` (управляет circuit breaker) | Да | | `manual` |
| changedBy | string | Идентификатор сотрудника, выполнившего переключение | Да | | |
| changedAt | string (ISO 8601) | Момент переключения | Да | | |
| transitionId | string (uuid) | Идентификатор записи в журнале переключений | Да | | |

## Пример ответа

*В разделе приводится пример возвращаемого ответа.*

```json
{
  "mode": "DEGRADED",
  "control": "manual",
  "changedBy": "e.petrov",
  "changedAt": "2026-07-04T12:40:11Z",
  "transitionId": "f4d9b7a2-3c1e-4a8f-9b6d-7e5c2a1f0d3b"
}
```

## Логика работы

1. Валидация тела запроса: `mode` принимает только значения `degraded` / `auto`, `reason` обязателен (400 при нарушении). Авторизация и проверка роли внутреннего контура (401/403).
2. При `mode = degraded`:
   - система немедленно переводится в режим `DEGRADED` независимо от текущего состояния circuit breaker;
   - вызовы ML-сервиса прекращаются, вся выдача обслуживается fallback-движком;
   - автоматические переходы `DEGRADED → RECOVERY` блокируются, пока kill switch активен (таймер открытия не запускается).
3. При `mode = auto`:
   - kill switch снимается, управление возвращается circuit breaker;
   - breaker стартует из состояния `DEGRADED` и проходит штатный цикл восстановления `DEGRADED → RECOVERY → NORMAL` (см. [Резервный сценарий рекомендаций](../Резервный%20сценарий%20рекомендаций.md)) – мгновенного включения ML не происходит, возврат плавный, через пробы.
4. Переключение фиксируется в журнале переключений (`Инициатор = manual`, `Причина = manual`, комментарий – из поля `reason`, автор переключения – из токена сотрудника) и отправляется уведомление дежурной смене. При `mode = auto` фиксируется смена способа управления: режимы до/после в записи могут совпадать.
5. Метод идемпотентен: повторный запрос с тем же `mode` не создаёт нового переключения и возвращает текущее состояние (200); в `transitionId` при этом возвращается идентификатор последнего действующего переключения.
