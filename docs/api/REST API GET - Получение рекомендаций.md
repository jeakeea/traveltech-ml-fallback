## Описание

| Параметр | Значение |
|----------|----------|
| **URL\*** | `/api/v1/trips/{tripId}/recommendations` |
| **Метод HTTP\*** | `GET` |
| **Наименование метода\*** | Получение персонализированных рекомендаций (отели, активности, маршруты) для поездки. Единый контракт для всех режимов работы системы (основной / гибридный / резервный) |
| **Частичный запрос атрибутов\*** | Нет |
| **Постраничный вывод\*** | Да |
| **Возврат информации о количестве записей (count)\*** | Да |
| **Управление сортировками\*** | Нет – порядок позиций определяется релевантностью, рассчитанной системой |
| **Сортировка по умолчанию\*** | По релевантности (позиция в сформированной выдаче), по убыванию |
| **Способ авторизации\*** | JWT (token) пользователя |
| **Роли\*** | Участники поездки (роли `organizer`, `member` в поездке `{tripId}`) |
| **Ссылка на swagger** | – |
| **http-статус коды в ответе\*** | 200 – ок (в том числе в резервном режиме); Ошибки: 400, 401, 403, 404, 503 |

## Входные параметры\*

| Параметр | Тип | Описание | Расположение | Обязательный | Значение по умолчанию | Пример |
|----------|-----|----------|--------------|--------------|----------------------|--------|
| authorization | string | JWT (token) пользователя | Header | Да | | |
| tripId | string (uuid) | Идентификатор поездки, для которой запрашиваются рекомендации. Направление, даты и бюджет берутся из поездки | Path | Да | | `7b2e6a1c-...` |
| category | string | Категория рекомендаций: `hotels` / `activities` / `routes` / `all` | Query | Нет | `all` | `hotels` |
| page | integer | Номер возвращаемой «страницы», начиная с 0 | Query | Нет | `0` | `0` |
| size | integer | Количество элементов на странице, максимум 50 | Query | Нет | `20` | `20` |

## Пример запроса

## Логика работы

Полное описание архитектуры, режимов и параметров – в документе [Резервный сценарий рекомендаций](../Резервный%20сценарий%20рекомендаций.md). Алгоритм метода:

1. Валидация запроса (400 при некорректных `tripId`, `category`, `page`, `size`) и авторизация (401); проверка, что поездка `{tripId}` существует (404) и пользователь – её участник (403).
2. Если `page > 0` и в снапшоте выдачи (TTL 15 минут, ключ: пользователь + поездка + категория) есть сформированная выдача – страница возвращается из снапшота без нового расчёта. Это обеспечивает консистентность пагинации; снапшот фиксирует уже отданный ответ и не является кэшем предсказаний наперёд.
3. Оркестратор проверяет состояние circuit breaker:
   - `NORMAL` – параллельно запускаются вызов ML-сервиса (таймаут **3000 мс**) и расчёт резервной выдачи fallback-движком;
   - `DEGRADED` – ML не вызывается, выдача строится fallback-движком (`meta.fallbackReason = circuit_open` или `manual` при kill switch);
   - `RECOVERY` – 10 % запросов идут по схеме `NORMAL` (проба), остальные – по схеме `DEGRADED`.
4. Сборка итоговой выдачи (режим `NORMAL` / проба):
   - ML ответил вовремя и confidence всех позиций ≥ 0,6 → выдача ML (`source = ml`);
   - есть позиции с confidence < 0,6, но их доля ≤ 50 % → слабые позиции заменяются верхними позициями резерва без дублей, сильные сохраняют места (`source = hybrid`, `fallbackReason = low_confidence`);
   - доля слабых позиций > 50 % → полностью резервная выдача (`source = fallback`, `fallbackReason = low_confidence`);
   - таймаут 3000 мс или ошибка ML → готовая резервная выдача (`source = fallback`, `fallbackReason = timeout | ml_error`); ошибка фиксируется в circuit breaker.
5. Резервная выдача рассчитывается только из внутренних данных (история, профиль, обезличенные агрегаты популярности) с жёсткими фильтрами по направлению, датам, доступности и бюджету поездки; предсказания модели не кэшируются и не переиспользуются.
6. Формируется ответ единого контракта; `meta.nature` определяет заголовок виджета на клиенте. Первая страница сохраняет снапшот выдачи для пагинации.
7. Записываются метрики (источник, латентности, confidence) и лог выдачи.
8. Если недоступны и хранилища резерва – возвращается статический топ направлений из конфигурации (`source = fallback`, `fallbackReason = reserve_unavailable`, `nature = popular`); при недоступности всех уровней – `503` с заголовком `Retry-After`, клиент скрывает блок рекомендаций без ошибки на экране.

## Коды ответа\*

| HTTP-статус | Код сообщения | Сообщение | Сценарий возникновения |
|-------------|---------------|-----------|------------------------|
| 200 | OK | Операция выполнена успешно | Выдача сформирована – в любом режиме, включая резервный: отказ ML, таймаут и низкий confidence не являются ошибками для клиента |
| 400 | BadRequest | Ошибка в синтаксисе запроса | Некорректный формат `tripId`, недопустимое значение `category`, `size` вне диапазона 1–50 |
| 401 | Unauthorized | Пользователь не авторизован | Токен отсутствует, невалиден или истёк |
| 403 | Forbidden | Нет прав для доступа к функции | Пользователь не является участником поездки `{tripId}` |
| 404 | NotFound | Поездка не найдена | Поездка с идентификатором `{tripId}` не существует или удалена |
| 503 | ServiceUnavailable | Сервис временно недоступен | Недоступны все уровни деградации, включая статический топ. Возвращается заголовок `Retry-After`; клиент скрывает блок рекомендаций |
