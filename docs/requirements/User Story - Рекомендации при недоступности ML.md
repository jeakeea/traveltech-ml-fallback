## Контекст

Сервис TravelTech показывает пользователю персонализированные рекомендации (отели, активности, маршруты) на основе ML-модели. ML-сервис может отказывать, отвечать дольше 3 секунд или возвращать предсказания с низкой уверенностью (confidence). Во всех этих случаях пользователь должен продолжать получать полезную выдачу без ошибок, пустых блоков и заметных задержек. Резервная выдача строится только на внутренних данных: истории пользователя, его профиле и обезличенной популярности.

Архитектура, режимы и параметры резервного сценария описаны в основном документе: [Резервный сценарий рекомендаций](../Резервный%20сценарий%20рекомендаций.md).

## Описание пользовательской истории

*Обязательный к заполнению раздел.*

| | |
|---|---|
| **Я как** | путешественник, планирующий поездку |
| **Хочу** | получать подборку рекомендаций (отели, активности, маршруты) даже когда ML-сервис недоступен, отвечает медленно или неуверенно |
| **Для того, чтобы** | продолжать планирование поездки без прерываний, ошибок и пустых экранов |

## Критерии приёмки

*Обязательный к заполнению раздел.*

| № | Если `<Предусловие>` | Когда `<Действие>` | Тогда `<Результат>` |
|---|----------------------|--------------------|---------------------|
| **Штатная выдача (ML доступен, уверенность высокая)** | | | |
| 1.1 | система в основном режиме **и** ML-сервис отвечает быстрее 3 секунд **и** confidence всех позиций ≥ порога (0,6) | пользователь открывает экран с блоком рекомендаций | система отображает ML-выдачу (`source = ml`) **и** параллельно рассчитанный резерв не используется **и** ответ соответствует единому контракту API |
| **Гибридная выдача (часть позиций с низким confidence)** | | | |
| 2.1 | система в основном режиме **и** ML-сервис вернул выдачу, где доля позиций с confidence < 0,6 не превышает 50 % | пользователь открывает экран с блоком рекомендаций | система заменяет слабые позиции лучшими позициями резервного расчёта без дублей (`source = hybrid`) **и** сильные ML-позиции сохраняют свои места **и** пользователь видит выдачу в том же виджете без каких-либо признаков замены |
| 2.2 | система в основном режиме **и** ML-сервис вернул выдачу, где доля позиций с confidence < 0,6 превышает 50 % | пользователь открывает экран с блоком рекомендаций | система отдаёт полностью резервную выдачу (`source = fallback`, `reason = low_confidence`) |
| **Резерв при задержке ML более 3 секунд** | | | |
| 3.1 | система в основном режиме **и** ML-сервис не ответил за 3000 мс | пользователь открывает экран с блоком рекомендаций | система немедленно отдаёт параллельно рассчитанную резервную выдачу (`source = fallback`, `reason = timeout`) **и** суммарное время ответа не превышает таймаут более чем на время сборки **и** таймаут учитывается как ошибка в circuit breaker |
| **Резерв при отказе ML-сервиса** | | | |
| 4.1 | система в основном режиме **и** ML-сервис возвращает ошибку или недоступен | пользователь открывает экран с блоком рекомендаций | система отдаёт резервную выдачу (`source = fallback`, `reason = ml_error`) **и** пользователь не видит сообщения об ошибке **и** ошибка учитывается в circuit breaker |
| **Автоматический переход в резервный режим** | | | |
| 5.1 | в скользящем окне 60 с доля ошибок ML ≥ 50 % при ≥ 10 запросах **или** произошло 5 ошибок подряд | наступает событие, достигающее порога | система переводит режим `NORMAL → DEGRADED` **и** прекращает вызовы ML-сервиса **и** фиксирует переключение в журнале (причина, время, инициатор – автоматика) **и** все последующие запросы обслуживаются резервом с p95 ≤ 300 мс |
| **Восстановление основного режима** | | | |
| 6.1 | система в режиме `DEGRADED` **и** истёк таймер открытия (30 с, при повторных провалах ×2 до 300 с) | наступает момент истечения таймера | система переходит в режим `RECOVERY` **и** направляет в ML 10 % запросов как пробы, остальные обслуживает резервом |
| 6.2 | система в режиме `RECOVERY` **и** за 120 с накоплено ≥ 30 проб с долей успеха ≥ 95 % | наступает событие достижения критерия | система возвращается в режим `NORMAL` **и** фиксирует переключение в журнале |
| 6.3 | система в режиме `RECOVERY` **и** доля ошибок на пробах ≥ 20 % | наступает событие достижения порога | система возвращается в режим `DEGRADED` **и** таймер открытия удваивается (не более 300 с) |
| **Холодный старт в резерве** | | | |
| 7.1 | резервный расчёт активен **и** у пользователя нет истории (менее 3 событий) **и** профиль не заполнен | пользователь открывает экран с блоком рекомендаций | система строит выдачу на обезличенной популярности по направлению и сезону **и** заголовок виджета – «Популярно у путешественников» (`nature = popular`) |
| **Групповая поездка в резерве** | | | |
| 8.1 | резервный расчёт активен **и** поездка групповая **и** пересечение интересов участников непустое | участник группы открывает экран с блоком рекомендаций | система строит выдачу по пересечению интересов участников **и** бюджетные предпочтения группы – пересечение бюджетных диапазонов из профилей участников (участвуют в скоринге) **и** жёсткий бюджетный фильтр – бюджет поездки |
| 8.2 | резервный расчёт активен **и** поездка групповая **и** пересечение интересов участников пустое | участник группы открывает экран с блоком рекомендаций | система использует профиль организатора поездки (интересы и бюджетный диапазон) |
| **Недоступность резервных данных** | | | |
| 9.1 | недоступны хранилища резерва (агрегаты популярности, профили) | пользователь открывает экран с блоком рекомендаций | система отдаёт статический топ направлений из конфигурации (уровень 3 иерархии деградации) |
| 9.2 | недоступен и статический уровень | пользователь открывает экран с блоком рекомендаций | сервис отвечает `503` **и** клиент скрывает блок рекомендаций **и** остальная страница остаётся работоспособной **и** пользователь не видит сообщения об ошибке |
| **Непрерывность UX при переключениях** | | | |
| 10.1 | пользователь просматривает уже отображённую выдачу | режим системы меняется (в любую сторону) | отображённая выдача не перерисовывается **и** новый режим применяется только к следующим запросам |
| 10.2 | пользователь листает многостраничную выдачу | пользователь запрашивает следующую страницу в пределах 15 минут | страницы читаются из снапшота выдачи первой страницы **и** позиции не перемешиваются между страницами даже при смене режима |
| 10.3 | выдача построена с участием истории или профиля (в любом режиме) | пользователь видит блок рекомендаций | заголовок блока – «Подобрано для вас» (`nature = personalized`); заголовок зависит только от природы данных, не от технического режима |

## Атрибуты полей форм

*Обязательный к заполнению раздел, если спецификация затрагивает формы пользовательского интерфейса.*

Полей ввода функциональность не содержит – виджет рекомендаций только отображает данные. Атрибуты отображаемых элементов:

| № | Наименование элемента | Описание | Тип, формат, ограничения | Условия отображения/Видимость |
|---|----------------------|----------|--------------------------|------------------------------|
| **Виджет рекомендаций** | | | | |
| 1 | Заголовок блока | Заголовок виджета рекомендаций | Строка: «Подобрано для вас» при `nature = personalized`; «Популярно у путешественников» при `nature = popular` | Отображается всегда, когда блок виден |
| 2 | Карточка позиции | Элемент выдачи: наименование, категория, цена, рейтинг | Данные позиции из ответа API; служебные поля `itemSource`, `confidence`, `score` пользователю не отображаются | Для каждой позиции текущей страницы выдачи |
| 3 | Блок рекомендаций | Контейнер виджета на странице планирования | – | Скрывается целиком при ответе `503` (уровень 4 деградации); сообщение об ошибке пользователю не отображается |
