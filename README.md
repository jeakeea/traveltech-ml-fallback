# TravelTech: резервный сценарий персонализированных рекомендаций

Спецификация механизма управляемой деградации функции рекомендаций для сервиса планирования путешествий. Отвечает на вопрос: что происходит с выдачей рекомендаций (отели, активности, маршруты), когда ML-модель отказывает, отвечает дольше 3 секунд или возвращает предсказания с низкой уверенностью – и как система возвращается в основной режим, когда модель оживает.

Решение кейса «Опять сломали ML». Документация оформлена в стиле docs-as-code: спецификации в Markdown, диаграммы как код (PlantUML).

## Идея решения

Вместо простого «выключателя» – два контура защиты и иерархия деградации из четырёх уровней. Пользователь ни на одном уровне не видит ошибку: меняется только степень персонализации выдачи.

![Компонентная архитектура](docs/diagrams/png/01%20-%20Компонентная%20архитектура.png)

**Контур запроса.** Каждый запрос рекомендаций параллельно запускает вызов ML-сервиса (таймаут 3000 мс) и расчёт резервной выдачи rule-based движком. К моменту любого исхода – таймаут, ошибка, низкая уверенность – резервная выдача уже готова, ответ пользователю не задерживается. Параллельный расчёт не перестраховка, а несущий элемент: он же питает **гибридную сборку** – если у части позиций ML-выдачи confidence ниже порога (0,6), заменяются только они, сильные позиции сохраняют свои места. Полный переход на резерв происходит лишь когда слабых позиций больше половины.

**Контур режима.** Circuit breaker с тремя состояниями (`NORMAL` → `DEGRADED` → `RECOVERY`) снимает нагрузку с отказавшего ML-сервиса и делает переключение плавным. Пороги асимметричны – гистерезис: в резерв система уходит за секунды (50 % ошибок за минуту или 5 подряд), а возвращается осторожно – через пробы на 10 % трафика со статистически значимым критерием успеха. Для дежурного инженера предусмотрен kill switch – принудительное включение резерва отдельным API-методом.

![Диаграмма состояний](docs/diagrams/png/02%20-%20Диаграмма%20состояний%20режимов.png)

**Иерархия деградации.**

| Уровень | Источник выдачи |
|---------|-----------------|
| 1 | ML-выдача – полная или гибридная |
| 2 | Rule-based резерв: история пользователя + профиль + обезличенная популярность |
| 3 | Статический топ направлений из конфигурации |
| 4 | Блок рекомендаций тихо скрывается; страница остаётся работоспособной |

**Резерв – только на внутренних данных.** Резервный скоринг взвешивает три источника (история, профиль, агрегаты популярности) и деградирует каскадно: у нового пользователя без истории выдача строится на популярности по направлению, у групповой поездки интересы берутся пересечением интересов участников. Предсказания модели принципиально не кэшируются и не переиспользуются – интересы быстро устаревают; резерв всегда рассчитывается в момент запроса по актуальным данным. Предрассчитываются только обезличенные агрегаты спроса – это статистика, а не выход модели.

**Честность к пользователю без демонстрации сбоев.** Заголовок виджета определяется природой данных, а не техническим состоянием системы: персональная выдача (в любом режиме) подписана «Подобрано для вас», неперсональная – «Популярно у путешественников». Контракт API един для всех режимов; служебные поля `source` / `fallbackReason` видит только аналитика.

## Диаграммы

Семь диаграмм (компонентная, состояния, три sequence, activity, модель данных) ведутся как код: PlantUML-блоки внутри документов и те же исходники отдельными файлами в [docs/diagrams/](docs/diagrams). Готовые рендеры – в [docs/diagrams/png/](docs/diagrams/png), они же встроены в документы.

Перегенерация после правок:

```bash
plantuml -charset UTF-8 -o png docs/diagrams/*.puml
```
